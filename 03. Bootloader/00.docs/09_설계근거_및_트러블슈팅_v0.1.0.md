- 프로젝트: NU-HC UART-BLE 부트로더
- 문서 버전: v0.1.0
- 기준일: 2026-06-18
- 작성자: 이한서

## 1. 문서 목적

본 문서는 부트로더 구현에서 내린 주요 설계 결정의 근거와, 개발·디버깅 과정에서 확인된 문제·해결책을 기록한다. "왜 이렇게 설계했는가"와 "어떤 함정을 어떻게 피하는가"를 다음 개발자에게 전달하는 것이 목적이다.

## 2. 설계 근거 (Design Rationale)

### 2.1 보레이트 19200 선택

| 항목 | 내용 |
| --- | --- |
| 결정 | UART 속도를 19200 bps로 고정 |
| 근거 | ATmega1281 16 MHz에서 115200은 `U2X=1`이어도 약 +2.1% 오차로 비트 샘플링이 어긋나 262 byte 스트림이 깨짐. 19200(`UBRR=51`)은 오차 0.2%로 안정 |
| 영향 | 모든 UART 구간(Host, NU-HC, MCU)을 19200으로 통일 |

### 2.2 인터럽트 + 원형큐 수신

| 항목 | 내용 |
| --- | --- |
| 결정 | 폴링 대신 인터럽트 + 256 byte 원형큐 |
| 근거 | 폴링 방식은 262 byte 연속 수신 중 오버런으로 byte를 유실. ISR이 즉시 `UDR`을 비워 큐에 적재하면 유실 방지 |
| 구현 | 채널별 RX/TX 큐 분리, `uint8_t` 인덱스로 AVR 원자적 접근 |

### 2.3 TX도 인터럽트화

| 항목 | 내용 |
| --- | --- |
| 결정 | 송신도 `UDRIE` 인터럽트로 처리 |
| 근거 | 폴링 송신은 byte당 CPU를 점유. 인터럽트 송신은 큐에 넣고 즉시 복귀 |
| 핵심 | `UDRIE`는 송신 데이터 있을 때만 set, 큐 비면 ISR이 clear → 무한 호출 방지 |

### 2.4 START_CODE 재동기

| 항목 | 내용 |
| --- | --- |
| 결정 | 패킷 수신 시 `0xEF` 찾을 때까지 선행 byte 폐기 |
| 근거 | 핸드셰이크 잔재('R', 'S' 등)나 노이즈가 첫 패킷 앞에 끼면 정렬이 밀려 CRC 실패. 재동기로 자동 복구 |
| 효과 | 정상 경로는 첫 byte가 `0xEF`라 1회 통과, 비정상 시에만 폐기 동작 |

### 2.5 청크 전송

| 항목 | 내용 |
| --- | --- |
| 결정 | 262 byte를 20 byte 단위로 분할, 20 ms 간격 송신 |
| 근거 | NU-HC는 UART→BLE를 182 byte 또는 16 ms마다 flush. 한 번에 보내면 마지막 자투리가 buffer에 갇히거나 단편화 |
| 효과 | flush 주기에 맞춰 마지막 byte까지 안정 전달 |

### 2.6 인밴드 핸드셰이크 (DTR 대체)

| 항목 | 내용 |
| --- | --- |
| 결정 | DTR 물리 리셋 대신 `'S'`/`'R'` 인밴드 신호 |
| 근거 | DTR 신호는 BLE 링크를 통과하지 못해 원격 MCU를 리셋할 수 없음 |
| 절차 | Host `'S'` 반복 → MCU `'R'` 응답 → 전송 시작 (리셋은 수동) |

### 2.7 packed 구조체 + little-endian

| 항목 | 내용 |
| --- | --- |
| 결정 | 패킷 구조체 `__attribute__((packed))`, 다중 byte는 little-endian |
| 근거 | 패딩이 있으면 `sizeof != 262`로 필드 위치 어긋남. x86 호스트와 AVR이 동일 byte 순서로 직렬화해야 일치 |

## 3. 트러블슈팅 (Known Issues)

### 3.1 BLE 미연결 시 ERROR 응답

| 항목 | 내용 |
| --- | --- |
| 증상 | 전송 시 호스트에 `0x0D 0x0A 45 52 52 4F 52`(`\r\nERROR`) 반복 수신 |
| 원인 | NU-HC가 Command Mode 상태에서 raw 패킷을 AT 명령으로 해석 |
| 해결 | 전송 전 BLE 연결 + Throughput Mode 진입 확인 |
| 주의 | `ERROR`의 `R`(0x52)을 핸드셰이크 'R'로 오인하지 않도록 주의 |

### 3.2 포트 점유 충돌

| 항목 | 내용 |
| --- | --- |
| 증상 | `write`는 성공하나 데이터가 도달하지 않음 |
| 원인 | 시리얼 터미널(Eclipse 등 `java` 프로세스)이 같은 포트 점유 |
| 진단 | `sudo lsof /dev/ttyUSB0` |
| 해결 | 터미널 연결 해제 후 송신 도구 단독 실행 |

### 3.3 DTR 자동 토글

| 항목 | 내용 |
| --- | --- |
| 증상 | 송신 도구 실행 시 BLE 연결이 끊김 |
| 원인 | `open()`이 USB-UART의 DTR을 자동 토글 (`DTR_Reset` 호출 안 해도 발생) |
| 해결 | `HUPCL`/`CLOCAL` 설정 또는 DTR 선 분리, 인밴드 핸드셰이크 사용 |

### 3.4 little-endian 착시

| 항목 | 내용 |
| --- | --- |
| 증상 | CRC 마지막 byte가 안 오는 것처럼 보임 |
| 원인 | CRC `0xF404`는 little-endian으로 `04 F4` 전송. `F4 04`를 기대해 마지막 `04`가 누락된 것으로 오해 |
| 해결 | 끝이 `04 F4`로 끝나는 것이 정상. byte 순서 확인 후 판단 |

### 3.5 TX ISR 인덱스 오타

| 항목 | 내용 |
| --- | --- |
| 증상 | 수신 정상, UART1 송신만 깨짐 |
| 원인 | `USART1_UDRE_vect`에서 `tx1_buf[tx0_tail]` 사용 (복붙 오타) |
| 해결 | `tx1_buf[tx1_tail]`로 수정 |

### 3.6 USART_Receive break 안티패턴

| 항목 | 내용 |
| --- | --- |
| 증상 | 큐가 비었을 때 쓰레기 값 반환 |
| 원인 | `while(head==tail) break;` — 비어도 통과 |
| 해결 | `while(head==tail);` 블로킹 대기 |

### 3.7 RxFlush의 인터럽트 복원

| 항목 | 내용 |
| --- | --- |
| 증상 | 핸드셰이크 후 'R'을 못 받거나 수신 멈춤 |
| 원인 | RxFlush가 인터럽트 꺼진 상태에서 호출되어 복원이 안 됨 |
| 해결 | `sreg=SREG; cli(); ...; SREG=sreg;`로 원래 인터럽트 상태 복원. `main`의 `sei()`가 RxFlush보다 먼저 실행되어야 함 |

### 3.8 F_CPU 심볼 손상

| 항목 | 내용 |
| --- | --- |
| 증상 | 빌드 시 `16000000: No such file or directory` 에러 |
| 원인 | Toolchain Symbols에 `F_CPU=16000000UL`이 분리되어 `16000000`만 입력값으로 전달 |
| 해결 | Symbols에 `F_CPU=16000000UL`을 한 덩어리로 정확히 입력 |

### 3.9 RxFlush 매 패킷 호출 금지

| 항목 | 내용 |
| --- | --- |
| 증상 | 연속 전송 시 중간 패킷 유실 가능 |
| 원인 | `recv_packet` 시작마다 RxFlush 호출 → 큐에 쌓인 다음 패킷 폐기 |
| 해결 | RxFlush는 핸드셰이크 직후 1회만. 패킷 수신 정렬은 START_CODE 재동기가 담당 |

## 4. 디버깅 방법론

| 기법 | 용도 |
| --- | --- |
| HEX 덤프 | 수신 raw byte를 16진수로 출력해 데이터 무결성·정렬 확인 |
| 단계 마커 | `'1'/'2'/'3'` 등을 출력해 코드가 어느 단계까지 도달했는지 확인 |
| LED 표시 | `PORTC`로 MCU 상태·진행 가시화 |
| 변수 격리 | 한 번에 한 변수만 변경해 원인 분리 |
| 채널 분리 | UART0(디버그)와 UART1(데이터)을 분리해 간섭 없이 관찰 |
