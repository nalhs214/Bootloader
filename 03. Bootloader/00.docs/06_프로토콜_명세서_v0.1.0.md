**Host to MCU 펌웨어 페이지 전송 및 ACK 핸드셰이크 프로토콜**

## 1. 문서 목적

본 문서는 Linux 호스트의 송신 도구(`bl_sender`)가 펌웨어 바이너리를 BLE 투명 전송 구간을 거쳐 대상 MCU(ATmega1281) 부트로더로 전달하기 위한 메시지 구조와 절차를 정의한다.

본 시스템에서 Host는 펌웨어를 페이지 단위로 분할·송신하는 장치이며, MCU는 수신한 페이지를 검증·기록하고 결과를 응답하는 장치이다. 두 NU-HC(nRF54L05) 모듈이 BLE Throughput Mode로 데이터를 변형 없이 중계한다.

```text
Linux Host (bl_sender)
        │ UART, 양방향
        ▼
NU-HC #1 (BLE Central)
        │ BLE, Throughput 투명 전송
        ▼
NU-HC #2 (BLE Peripheral)
        │ UART, 양방향
        ▼
ATmega1281 Bootloader
```

---

## 2. 기본 전제

- Host는 MCU로 페이지 데이터를 송신하고, MCU는 페이지마다 ACK 또는 NAK를 응답한다.
- 본 프로토콜은 Request/Response가 아닌 데이터 전송 + 응답(ACK/NAK) 구조이다.
- 전송 시작 전 Host와 MCU는 `'S'`/`'R'` 핸드셰이크로 동기화한다.
- 거리 결과 프로토콜과 달리, 본 프로토콜은 재전송을 수행한다(NAK/타임아웃 시 최대 3회).
- 프레임 손상 여부는 CRC16으로 판단한다.
- 손상된 프레임에 대해 MCU는 NAK를 응답하고, Host는 재전송한다.
- 한 페이지(256 byte)는 1개 패킷(262 byte)으로 전송하며 분할하지 않는다.
- 다중 바이트 정수값은 little-endian으로 저장한다.

---

## 3. 참고 메시지 구조

기존 UART 메시지 정의 문서의 기본 구조인 `Start Code / Type / Data Length / Data` 형식을 따른다. 단, 본 프로토콜은 고정 256 byte 데이터 페이지를 사용하므로 `DATA_LENGTH`를 2 byte로 확장한다.

| Byte | 필드 | 크기 | 설명 |
| --- | --- | --- | --- |
| 0 | START_CODE | 1 byte | 메시지 시작 코드 |
| 1 | MSG_TYPE | 1 byte | 메시지 종류 |
| 2~3 | DATA_LENGTH | 2 byte | DATA 영역 길이 (little-endian) |
| 4~N | DATA | 256 byte | 펌웨어 페이지 데이터 |
| N+1~N+2 | CRC16 | 2 byte | 무결성 검증 (little-endian) |

---

## 4. UART 메시지 기본 구조

| 필드 | 크기 | 값 | 설명 |
| --- | --- | --- | --- |
| START_CODE | 1 byte | `0xEF` | 메시지 시작 코드 |
| MSG_TYPE | 1 byte | 메시지별 정의 | 메시지 종류 |
| DATA_LENGTH | 2 byte | `0x0100` (256) | DATA 영역의 byte 길이 |
| DATA | 256 byte | 메시지별 정의 | 펌웨어 페이지 (부족분 `0xFF` 패딩) |
| CRC16 | 2 byte | 계산값 | CRC16 검증값 |

- 전체 메시지 길이 = `1 + 1 + 2 + 256 + 2 = 262 byte` (고정)
- 구조체는 packed 정렬하여 `sizeof = 262`를 보장한다.

---

## 5. Byte Order

모든 2 byte 이상 정수값은 little-endian으로 저장한다.

```c
uint16_t crc = 0xF404;
// UART 전송 순서 = 0x04 0xF4
```

`DATA_LENGTH = 256 (0x0100)`은 `00 01` 순서로 저장된다. 수신 측은 `raw[2] | (raw[3] << 8)`로 복원한다.

---

## 6. 메시지 타입 정의

| 메시지 이름 | 값 | 방향 | 설명 |
| --- | --- | --- | --- |
| `MSG_DATA` | `0x01` | Host → MCU | 펌웨어 데이터 페이지 |
| `MSG_END` | `0x02` | Host → MCU | 전송 종료, 응용 점프 요청 |
| `ACK` | `0x06` | MCU → Host | 정상 수신·기록 응답 (단일 byte) |
| `NAK` | `0x15` | MCU → Host | 오류, 재전송 요구 응답 (단일 byte) |
| `'S'` | `0x53` | Host → MCU | 핸드셰이크 시작 신호 (단일 byte) |
| `'R'` | `0x52` | MCU → Host | 핸드셰이크 준비 응답 (단일 byte) |

`ACK`/`NAK`/`'S'`/`'R'`은 프레임 구조 없이 단일 byte로 송수신한다.

---

## 7. MSG_DATA (0x01)

### 7.1 설명

`MSG_DATA`는 펌웨어 한 페이지(256 byte)를 MCU로 전달하는 메시지이다. Host는 페이지마다 이 메시지를 1회 송신하고, MCU의 ACK를 받은 뒤 다음 페이지로 진행한다.

### 7.2 프레임 구조

```text
START_CODE  = 0xEF
MSG_TYPE    = 0x01
DATA_LENGTH = 0x0100  (256, little-endian → 00 01)
```

전체 UART 메시지 길이 = `1 + 1 + 2 + 256 + 2 = 262 byte`

### 7.3 DATA 구조

| Offset | 필드 | 크기 | 자료형 | 설명 |
| --- | --- | --- | --- | --- |
| 0 | `start_code` | 1 | `uint8_t` | `0xEF` |
| 1 | `msg_type` | 1 | `uint8_t` | `0x01` |
| 2 | `data_length` | 2 | `uint16_t` | `256`, little-endian |
| 4 | `data` | 256 | `uint8_t[256]` | 펌웨어 페이지 (부족분 `0xFF`) |
| 260 | `crc16` | 2 | `uint16_t` | CRC16, little-endian |

---

## 8. MSG_END (0x02)

### 8.1 설명

`MSG_END`는 모든 페이지 전송이 끝났음을 알리고 MCU에 응용 프로그램 점프를 요청하는 메시지이다. 프레임 길이를 `MSG_DATA`와 동일하게 262 byte로 통일하며, `data` 영역은 `0xFF`로 채운다.

### 8.2 프레임 구조

```text
START_CODE  = 0xEF
MSG_TYPE    = 0x02
DATA_LENGTH = 0x0000  (0, little-endian → 00 00)
data        = 0xFF × 256
```

MCU는 `MSG_END` 수신·검증 후 ACK를 보내고 `jump_to_app()`을 실행한다.

---

## 9. 응답 메시지 (ACK / NAK)

| 응답 | 값 | 조건 |
| --- | --- | --- |
| `ACK` | `0x06` | START_CODE 정렬 + CRC 일치 + 페이지 기록 성공 |
| `NAK` | `0x15` | CRC 불일치, 또는 부트 영역(0x1E000 이상) 기록 요청 |

응답은 MCU의 UART1을 통해 NU-HC #2 → BLE → NU-HC #1 → Host로 역방향 전달된다.

---

## 10. 핸드셰이크 절차

전송 시작 전, Host와 MCU는 다음 순서로 동기화한다.

```text
Host                                   MCU
 │  'S' 반복 송신 (200 ms 주기)  ─────▶ │  부트 대기 중 'S' 수신
 │                                      │  USART1_RxFlush() (RX 잔재 제거)
 │ ◀──────────────  'R' 응답            │  USART1_Transmit('R')
 │  'R' 수신 → 패킷 전송 시작            │  USART1_FlushTx() 후 run_bootloader
```

핸드셰이크는 DTR 물리 리셋이 BLE 링크를 통과하지 못하는 제약을 인밴드 신호로 해결한다.

---

## 11. START_CODE 재동기

MCU는 패킷 수신 시작 시 `START_CODE(0xEF)`를 만날 때까지 선행 byte를 폐기한다. 핸드셰이크 잔재나 노이즈가 앞에 끼어도 정렬을 복구하기 위한 안전망이다.

```c
do {
    raw[0] = USART1_Receive();
} while (raw[0] != START_CODE);   // 0xEF 찾을 때까지 폐기
for (uint16_t i = 1; i < PKT_SIZE; i++)
    raw[i] = USART1_Receive();    // 나머지 261 byte
```

정상 경로에서는 첫 byte가 곧 `0xEF`이므로 폐기 없이 1회에 통과한다.

---

## 12. CRC16 정의

`crc16`은 `START_CODE`와 `crc16` 필드를 제외한 영역에 대해 계산한다.

| 항목 | 포함 여부 |
| --- | --- |
| `START_CODE` | 제외 |
| `MSG_TYPE` | 포함 |
| `DATA_LENGTH` | 포함 |
| `DATA` (256 byte) | 포함 |
| `crc16` | 제외 |

계산 범위 = `MSG_TYPE(1) + DATA_LENGTH(2) + DATA(256) = 259 byte`

CRC 알고리즘은 **CRC-16/CCITT-FALSE**를 사용한다.

| 항목 | 값 |
| --- | --- |
| Polynomial | `0x1021` |
| Init | `0xFFFF` |
| XorOut | `0x0000` |
| ReflectIn | false |
| ReflectOut | false |

Host와 MCU는 동일한 `crc16()` 구현을 사용한다. MCU는 CRC가 맞지 않는 프레임에 NAK를 응답하고, Host는 재전송한다.

---

## 13. 재전송 정책

거리 결과 프로토콜과 달리, 본 프로토콜은 신뢰성 있는 펌웨어 전송을 위해 재전송을 수행한다.

| 상황 | Host 동작 |
| --- | --- |
| ACK(`0x06`) 수신 | 다음 페이지로 진행 |
| NAK(`0x15`) 수신 | 동일 페이지 재전송 |
| 응답 타임아웃 (약 5 s) | 동일 페이지 재전송 |
| 재전송 3회 초과 (`MAX_RETRY`) | 전송 실패 종료 |

---

## 14. 청크 전송

Host는 262 byte 패킷을 한 번에 보내지 않고, NU-HC의 flush 동작(182 byte 또는 16 ms)에 맞춰 작은 단위로 분할 송신한다.

| 항목 | 값 |
| --- | --- |
| 청크 크기 | 20 byte |
| 청크 간 대기 | 20 ms (`usleep`) |
| 청크 후 처리 | `tcdrain`으로 전송 완료 대기 |

청크 단위는 NU-HC가 마지막 자투리까지 BLE로 내보내도록 보장하기 위한 값이며, 안정 동작 확인 후 조정할 수 있다.

---

## 15. 예시 프레임 (MSG_DATA)

다음은 펌웨어 페이지 1건(첫 페이지)을 전송하는 예시이다.

| 필드 | 값 |
| --- | --- |
| `start_code` | `0xEF` |
| `msg_type` | `0x01` (MSG_DATA) |
| `data_length` | `256` (`0x0100`) |
| `data[0..3]` | `65 C0 00 00` (페이지 시작 예) |
| `crc16` | `0xF404` |

```text
EF 01 00 01
65 C0 00 00 6B C0 00 00 ... (256 byte)
04 F4
```

`data_length`는 little-endian으로 `00 01`, `crc16` `0xF404`는 `04 F4` 순서로 저장된다.

## 16. 전송 시퀀스 전체

```text
1. Host: bin 파일 로드 → 256 byte 페이지 N개 산출
2. Host ↔ MCU: 'S'/'R' 핸드셰이크
3. for page in 1..N:
     Host: MSG_DATA 패킷(262 byte) 청크 전송
     MCU : START_CODE 재동기 → 262 byte 수신 → CRC 검증
            ├ 성공: flash 기록 → ACK
            └ 실패: NAK
     Host: ACK → 다음 / NAK·타임아웃 → 재전송(≤3)
4. Host: MSG_END 전송
   MCU : ACK → jump_to_app()
```

## 17. UART 설정값

| 항목 | 값 |
| --- | --- |
| Baudrate | 19200 bps |
| Data bits | 8 |
| Parity | None |
| Stop bits | 1 |
| Flow control | 미사용 |
| Direction | 양방향 (데이터 + 응답) |

ATmega1281은 16 MHz 클럭에서 115200 bps 사용 시 약 +2.1% 오차로 비트 샘플링이 깨지므로, 오차 0.2%인 19200 bps(`UBRR=51`)를 사용한다.

## 18. 참고 구현 구조체

아래 구조체는 DATA 영역을 표현하기 위한 참고용이며, packing과 byte order는 실제 구현에서 검증한다.

```c
#include <stdint.h>

#define START_CODE   0xEF
#define MSG_DATA     0x01
#define MSG_END      0x02
#define ACK          0x06
#define NAK          0x15
#define PAGE_SIZE    256
#define PKT_SIZE     262      /* 1+1+2+256+2 */
#define BOOT_START   0x1E000UL

/**
 * @brief 부트로더 데이터 패킷 (262 byte).
 *
 * @details 모든 2 byte 이상 정수 필드는 little-endian으로 직렬화한다.
 *          packed 정렬로 sizeof = 262를 보장한다.
 */
typedef struct {
    uint8_t  start_code;        /* 0xEF */
    uint8_t  msg_type;          /* 0x01=DATA, 0x02=END */
    uint16_t data_length;       /* 256, little-endian */
    uint8_t  data[PAGE_SIZE];   /* 펌웨어 페이지 256 byte */
    uint16_t crc16;             /* CRC16, little-endian */
} __attribute__((packed)) bl_data_packet_t;
```
