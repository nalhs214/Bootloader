# 요구사항 명세서

- 프로젝트: NU-HC UART-BLE 부트로더
- 문서 버전: v0.1.0
- 기준일: 2026-06-18
- 작성자: 이한서

## 1. 제품 목표

본 제품은 하드웨어 디버거(J-Link 등) 없이, 원격 BLE 링크를 경유하여 대상 MCU(ATmega1281)의 응용 펌웨어를 무선으로 갱신하는 부트로더 시스템이다. Linux 호스트의 송신 도구가 펌웨어 바이너리를 페이지 단위 패킷으로 분할하여, 두 개의 NU-HC(nRF54L05) 모듈로 구성된 BLE 투명 전송 구간을 거쳐 대상 MCU의 부트로더에 전달한다. 부트로더는 각 페이지를 검증·기록하고 응답(ACK/NAK)으로 신뢰성을 보장한다.

## 2. 시스템 구성

| 구성요소 | 설명 |
| --- | --- |
| Host Sender | Linux 기반 송신 CLI(`bl_sender`), 바이너리 분할·패킷화·재전송 관리 |
| USB-UART | CP210x 어댑터, 호스트와 NU-HC #1 연결 |
| NU-HC #1 | nRF54L05 모듈, BLE Central, AT CMD 펌웨어, Throughput 투명 전송 |
| NU-HC #2 | nRF54L05 모듈, BLE Peripheral, AT CMD 펌웨어, Throughput 투명 전송 |
| Target MCU | ATmega1281, 부트로더 + 응용 펌웨어 적재 대상 |

## 3. 사용자 관점 요구사항

| ID | 요구사항 |
| --- | --- |
| UR-001 | 사용자는 하드웨어 디버거 연결 없이 응용 펌웨어를 갱신할 수 있어야 한다. |
| UR-002 | 사용자는 BLE 무선 링크를 통해 원격으로 펌웨어를 전송할 수 있어야 한다. |
| UR-003 | 사용자는 펌웨어 전송 성공 여부를 호스트 콘솔에서 확인할 수 있어야 한다. |
| UR-004 | 시스템은 전송 중 오류가 발생한 페이지를 자동으로 재전송해야 한다. |
| UR-005 | 시스템은 부트로더 영역을 덮어쓰지 않아야 한다. |
| UR-006 | 시스템은 펌웨어 갱신 후 응용 프로그램으로 자동 진입해야 한다. |
| UR-007 | 사용자는 별도 전용 도구 없이 표준 시리얼 포트와 CLI로 동작시킬 수 있어야 한다. |

## 4. 시스템 기능 요구사항

| ID | 요구사항 |
| --- | --- |
| SYS-F-001 | 호스트는 펌웨어 바이너리를 256바이트 페이지 단위로 분할해야 한다. |
| SYS-F-002 | 호스트는 각 페이지를 262바이트 고정 패킷으로 구성해야 한다. |
| SYS-F-003 | 패킷은 START_CODE(0xEF), MSG_TYPE, DATA_LENGTH, DATA(256), CRC16으로 구성된다. |
| SYS-F-004 | 호스트와 MCU는 동일한 CRC16(CCITT-FALSE)으로 무결성을 검증해야 한다. |
| SYS-F-005 | 호스트는 전송 전 MCU와 핸드셰이크(S/R)로 동기화해야 한다. |
| SYS-F-006 | MCU는 부트 진입 후 일정 시간 동안 호스트의 시작 신호를 대기해야 한다. |
| SYS-F-007 | MCU는 수신 패킷의 CRC가 일치할 때만 페이지를 기록해야 한다. |
| SYS-F-008 | MCU는 페이지 기록 성공 시 ACK, 실패 시 NAK를 응답해야 한다. |
| SYS-F-009 | MCU는 START_CODE를 찾을 때까지 선행 바이트를 폐기(재동기)해야 한다. |
| SYS-F-010 | MCU는 MSG_END 수신 시 응용 프로그램으로 점프해야 한다. |
| SYS-F-011 | MCU는 부트로더 영역(0x1E000 이상) 기록 요청을 거부해야 한다. |
| SYS-F-012 | 호스트는 ACK 수신 시 다음 페이지로, NAK/타임아웃 시 재전송으로 진행해야 한다. |
| SYS-F-013 | NU-HC 모듈은 수신 데이터를 변형 없이 투명 전송(Throughput Mode)해야 한다. |

| ID | 요구사항 | 초기값 |

| SYS-T-001 | UART 통신 속도 | 19200 bps (8N1) |

| SYS-T-003 | 호스트 ACK 대기 타임아웃 | 약 5 s |

| SYS-T-006 | 청크 전송 단위 | 20 바이트 (NU-HC flush 정렬) |

## 6. 제약 요구사항

| ID | 요구사항 |
| --- | --- |
| SYS-C-001 | 대상 MCU는 ATmega1281, 16 MHz, 5 V 기준으로 한다. |
| SYS-C-002 | 16 MHz 클럭 한계로 115200 bps는 사용하지 않고 19200 bps를 사용한다. |
| SYS-C-003 | NU-HC 모듈 펌웨어 갱신에는 chip erase를 수행하는 `west flash`를 사용하지 않는다. |
| SYS-C-004 | NU-HC 투명 전송은 182바이트 또는 16 ms 간격으로 flush되며, 청크 크기를 이에 맞춘다. |
| SYS-C-005 | DTR 물리 리셋은 BLE 링크를 통과하지 못하므로 인밴드 핸드셰이크로 동기화한다. |
| SYS-C-006 | 패킷 구조체는 packed로 정렬하여 sizeof가 정확히 262바이트여야 한다. |

## 7. 비기능 요구사항

| ID | 요구사항 |
| --- | --- |
| SYS-N-001 | MCU 수신부는 인터럽트 + 원형큐 구조로 오버런을 방지해야 한다. |
| SYS-N-002 | 다중 바이트 값(DATA_LENGTH, CRC16)은 little-endian으로 송수신해야 한다. |
| SYS-N-003 | MCU는 디버그 출력(UART0)과 데이터 채널(UART1)을 분리해야 한다. |
| SYS-N-004 | 시스템은 첫 패킷의 정렬 오류에도 재동기로 복구할 수 있어야 한다. |
| SYS-N-005 | 부트로더는 인터럽트 벡터 재배치(IVSEL)를 올바르게 처리해야 한다. |

## 8. 제품 범위

### 8.1 포함

- 페이지 단위 무선 펌웨어 전송
- CRC16 기반 무결성 검증
- ACK/NAK 재전송 신뢰성 보장
- S/R 핸드셰이크 및 START_CODE 재동기
- 갱신 후 응용 프로그램 자동 점프

### 8.2 제외

- 펌웨어 암호화 및 서명 검증(AES/SHA)
- 완전 무인 자동 리셋(현재는 수동 리셋 기반)
- nRF54L05 자체 포팅(장기 목표, 별도 단계)
- 롤백/이중 슬롯(A/B) 갱신

## 9. 성공 기준

1. `LED_blink.bin`(5페이지)이 BLE 경로로 누락 없이 전송된다.
2. 모든 페이지에 대해 ACK가 수신된다.
3. MSG_END 후 응용 프로그램으로 점프하여 LED 동작이 확인된다.
4. 첫 패킷 정렬 오류 시 재동기로 복구된다.
5. CRC 오류 페이지는 NAK 후 재전송으로 복구된다.
